# R34 Video Watch Archive Downloader 项目介绍与开发方向建议

## 项目定位

R34 Video Watch Archive Downloader 是一个面向 `rule34video.com` 的 Tampermonkey / Userscript 工具。它不是通用视频下载器，也不是独立桌面下载软件，而是一个嵌入站点页面工作的轻量归档与批量下载面板。

项目的核心价值是把“浏览、记录、采集、解析、导出、下载”放在同一个网页上下文里完成。用户在站内浏览视频时，脚本可以自动识别已观看视频并加入队列；在列表页、搜索页、上传者页、用户页、收藏页等页面中，用户也可以手动采集当前页或自动翻页采集多页内容。脚本随后解析视频直链、清晰度和作品元信息，并支持浏览器直接下载、直链 TXT 导出、YT-DLP 命令 TXT 导出和元信息 JSONL 导出。

当前仓库维护英文和中文两个 UI 版本：

- `r34-video-watch-archive-downloader.user.js`：英文界面脚本。
- `r34-video-watch-archive-downloader.zh.user.js`：中文界面脚本。

两个脚本应保持功能逻辑一致，差异主要限于 userscript 元数据和可见 UI 文案。仓库中的 `README.md`、`README_ZH.md`、`Intro.MD`、`Intro_ZH.MD` 和截图素材用于发布与说明，`Agent.md` 用于约束后续维护者或编程智能体。

## 当前能力概览

项目当前已经具备一个完整的站内下载工作流：

- 页面注入悬浮控制面板，支持拖动、折叠和高级选项。
- 自动监听真实页面、AJAX 页面、伪跳转和 DOM 变化，识别已观看视频页。
- 支持当前页采集和多页采集，并能处理分页、页码回绕和重复检测。
- 维护持久化队列、已见记录、下载统计和用户设置。
- 支持清晰度选择：最佳、8K、4K、1080p、720p、480p、360p。
- 解析视频直链、原始文件名、标题、ID、标签、上传者、统计数据等元信息。
- 浏览器下载模式可先保存元信息 JSON，再下载视频本体。
- 支持导出直链 TXT、YT-DLP 命令 TXT、元信息 JSONL。
- 下载队列支持并发提交、进度显示、失败保留、手动重试和刷新后的状态恢复。
- 文件名可以按 ID、标题、原文件名组合生成。

这个设计更接近“站内归档助手”而不是单纯的“下载按钮”。它解决的是重复浏览、批量收集、元信息保留、失败重试和后续工具衔接的问题。

## Reference 项目观察

`Reference` 目录中包含的项目可以分为四类。

第一类是 Bilibili 用户脚本下载器，例如 `Bilibili DownloadX` 和 `injahow/bilibili视频下载`。这些脚本的特点是站点专用、注入页面 UI、读取页面或接口中的视频信息，并提供多种下载方式。`injahow/bilibili视频下载` 的功能面更宽，包含 Web、RPC、Blob、Aria、API 等模式，还支持字幕、弹幕、换源播放和配置面板。对本项目的启发是：下载方式可以插件化，外部下载器可以作为高级模式，但配置复杂度必须被控制。

第二类是 YouTube 用户脚本下载器，例如 `YouTube Ultimate Downloader`。它强调浮动按钮、多语言描述、站点 SPA 适配和外部下载服务跳转。对本项目的启发是：SPA 页面监听、按钮入口和多语言发布可以继续强化；但本项目不应依赖不透明的第三方转换站点，否则稳定性、隐私和可维护性都会下降。

第三类是站点体验增强脚本，例如 `主流视频网站 播放速度自由控制+记忆速度交换`。它说明轻量面板、拖拽、快捷操作、跨页面视频元素监听这些交互模式是用户脚本的优势。对本项目的启发是：面板应保持紧凑、可恢复、少打扰，并且对 AJAX / 动态 DOM 有较好的适应性。

第四类是 `参考项目.txt` 中列出的独立工具和扩展，包括 YouDownSet、Vidmost、bilidown、BBDown、DownKyi、RV、MagicPH 和 Video DownloadHelper。这些项目覆盖命令行、桌面 GUI、浏览器扩展和专站脚本，说明视频下载工具常见的差异点集中在：站点覆盖范围、清晰度/格式选择、批量采集、元数据保存、外部下载器集成、队列管理、错误恢复和用户配置复杂度。

## 网络同类项目观察

网上类似项目大致可以分成以下几类：

- 通用命令行核心：`yt-dlp` 是功能丰富的命令行音视频下载器，支持大量站点，并适合作为导出命令或外部处理链的目标。
- 专站命令行下载器：`BBDown` 面向 Bilibili，强调 URL / ID 解析、清晰度选择、多分 P、字幕、音视频合并和 aria2c 集成；但其主仓库已归档，说明专站下载器维护压力很高。
- 专站桌面 GUI：`DownKyi / downkyicore` 面向 Bilibili，提供清晰 UI、批量下载、8K / HDR / 杜比视界、音视频提取、去水印、字幕/弹幕等功能，适合重度用户。
- 浏览器扩展：`Video DownloadHelper` 面向多站点，用扩展权限捕获媒体并提供下载入口，覆盖面广，但需要安装扩展并处理浏览器平台限制。
- Rule34Video 专项工具：`trickerer01/RV` 是面向 `rule34video.com` 的高级命令行下载/抓取工具，重点在搜索过滤；`Gabriel712/r34_downloader` 是 Python 应用，通过 SQLite 记录避免重复下载；SleazyFork 上也存在 Rule34Video 链接提取类脚本。

与这些项目相比，本项目的差异化在于：它不追求覆盖所有站点，也不要求用户使用命令行或桌面客户端；它直接嵌入目标站点，利用用户当前浏览上下文完成归档、采集和下载任务。它的优势是轻量、低安装成本、站内流程顺手；短板是浏览器下载能力、跨会话可靠性、复杂批量任务和外部下载器集成能力天然弱于独立客户端。

## 可用于发布页的项目介绍

R34 Video Watch Archive Downloader 是一个用于 `rule34video.com` 的轻量级 Tampermonkey 脚本，面向需要整理已看视频、批量采集列表页视频、保存作品元信息并导出下载任务的用户。

脚本会在页面右下角提供一个可折叠、可拖动的悬浮面板。用户可以一键采集当前页面中的视频，也可以从搜索页、上传者页、用户页、收藏页等列表页开始连续采集多页内容。浏览视频详情页时，脚本能够自动识别已观看项目并加入队列，适合把日常浏览过程自然转化为可回溯的观看归档。

采集完成后，脚本会解析视频直链、可用清晰度和作品元信息。用户可以选择最佳、8K、4K、1080p、720p、480p 或 360p 清晰度，并按需要组合 ID、标题和原文件名生成下载文件名。下载方式支持浏览器直接下载，也支持导出直链 TXT、YT-DLP 命令 TXT 和元信息 JSONL，方便交给外部工具继续处理。

脚本支持中英文两个界面版本，功能逻辑保持一致。当前版本包含下载进度显示、队列统计、失败保留、手动重试、元信息 JSON 保存和刷新后的状态恢复，适合需要轻量归档、批量收集和可控下载流程的用户。

## 给开发者的方向建议

### 1. 保持项目边界清晰

本项目最有价值的定位是“Rule34Video 站内归档与下载助手”。不要过早扩展成通用视频下载器，也不要把桌面下载器、媒体合并、转码等重功能全部塞进 userscript。通用下载和复杂媒体处理应优先通过 YT-DLP、aria2c、FFmpeg 等外部工具衔接。

建议继续强化三个核心场景：

- 浏览时自动归档已看视频。
- 列表页批量采集和去重。
- 导出可被外部工具稳定消费的链接、命令和元信息。

### 2. 抽离站点解析、队列状态和 UI 文案

当前英文脚本和中文脚本需要保持功能同步，后续维护风险会随着代码体积增加而上升。建议逐步把代码拆成三个层次：

- 核心逻辑：队列、状态机、下载、导出、重试。
- 站点适配：URL 判断、分页识别、视频直链解析、元信息提取。
- 语言包：英文和中文 UI 文案、提示、按钮标题。

如果仍以单文件 userscript 发布，可以在构建阶段拼接生成中英文版本；如果暂时不引入构建工具，也至少应把两个脚本中的函数结构、常量命名和逻辑顺序保持严格一致，减少人工 diff 成本。

### 3. 优先完善可恢复性和可观测性

批量下载最容易出问题的地方不是“开始下载”，而是中途失败、刷新、重复提交和用户不知道发生了什么。建议继续加强：

- 每个任务的状态迁移记录，例如 pending、fetching、ready、downloading、done、failed。
- 失败原因分类，例如解析失败、无可用清晰度、GM_download 失败、元信息下载失败、页面刷新中断。
- 导出诊断日志，方便用户反馈和开发者复现。
- 队列快照导入/导出，避免浏览器存储被清理后无法恢复。

### 4. 把外部下载器作为高级工作流，而不是默认路径

参考 Bilibili 下载脚本、BBDown、DownKyi 和 yt-dlp，外部下载器能显著提升下载能力，但会增加配置门槛。建议采用分层策略：

- 默认模式：浏览器直接下载，尽量少配置。
- 稳定高级模式：导出 YT-DLP 命令和 JSONL 元信息。
- 可选高级模式：aria2c / RPC / 自定义 API 模板，仅在用户明确配置后启用。

这样可以保留轻量脚本的安装优势，同时给重度用户足够的扩展空间。

### 5. 强化元信息质量

本项目已经保存标题、ID、标签、上传者、清晰度、原文件名等信息。后续建议把元信息作为项目的重要资产来维护：

- 明确定义 JSONL 字段 schema，并在 README 中说明。
- 增加 `sourceUrl`、`postId`、`selectedQuality`、`availableQualities`、`collectedAt`、`downloadedAt`、`scriptVersion` 等稳定字段。
- 避免字段名频繁变动；必须变动时提供版本号。
- 对缺失字段做空值约定，方便后续脚本、表格或数据库导入。

这会让项目从“下载工具”进一步变成“个人媒体归档工具”。

### 6. 增加轻量测试和回归样例

用户脚本容易被站点 DOM 变化破坏。建议补充不依赖真实登录和真实下载的测试材料：

- 保存典型列表页、视频详情页、分页区块、伪跳转后的 HTML fixture。
- 为 URL 解析、分页识别、清晰度选择、文件名生成、元信息提取写纯函数测试。
- 为中英文脚本增加版本号、配置项、状态常量一致性检查。
- 对发布前流程增加 checklist：安装头、版本号、README、Intro、截图、SleazyFork 链接同步。

这类测试成本低，但能显著降低后续维护风险。

### 7. 谨慎处理合规和隐私边界

建议在 README 和发布页中明确提醒用户遵守当地法律、网站规则和版权要求。脚本不应上传用户浏览记录、下载记录或队列数据到第三方服务。若未来支持远程 API、RPC 或第三方下载服务，应在 UI 中明确说明数据会发送到哪里。

### 8. 短期路线图建议

短期优先级可以按以下顺序推进：

1. 补齐项目介绍文案、发布页文案和开发者维护说明。
2. 固化元信息 JSONL schema，并给出示例。
3. 增加队列导入/导出和诊断日志导出。
4. 把中英文脚本的差异检查自动化。
5. 抽离语言包或建立构建生成流程。
6. 增加 HTML fixture 和纯函数测试。
7. 设计 YT-DLP / aria2c 命令模板配置，让高级用户可扩展。

长期来看，本项目应坚持“小而稳定”的方向：浏览器内负责识别、采集、归档、排队和导出；复杂下载、合并、转码和跨站支持交给外部成熟工具。
